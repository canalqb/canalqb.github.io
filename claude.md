## ✍️ 3. REGRAS DE REDAÇÃO E HUMANIZAÇÃO (UX Writing & AEO)








### ✅ Sempre fazer:








- **Information Gain:** Adicione insights originais e testes reais. Nunca apenas reescreva o que já existe.
- **AEO (Answer Engine Optimization):** Use `<h2>` em formato de pergunta. Logo abaixo, responda em parágrafo de **40 a 60 palavras** (answer target direto).
- **SGE/AEO Block:** Imediatamente após o H1, inclua um bloco **TL;DR (Resumo Executivo)** com os 3 pontos fundamentais do post em bullet points.
Objetivo:
- Facilitar extração por mecanismos de IA e busca generativa (ChatGPT, Gemini, Perplexity, Claude etc.)
- Melhorar escaneabilidade
- Entregar contexto imediato ao leitor








Estrutura obrigatória:








TL;DR:
- [Insight principal]
- [Resultado, impacto ou benefício]
- [Conclusão prática ou recomendação]








Regras:
- Máximo de 3 bullet points
- Cada bullet deve ter entre 1 e 2 linhas
- Linguagem direta e objetiva
- Evite introduções, contexto excessivo e explicações longas
- Sempre que possível, inclua números, ganhos percentuais, tempo ou impacto real
- O TL;DR deve resumir o post inteiro, não apenas a introdução
- Escreva o TL;DR somente após concluir o conteúdo completo








Fórmula recomendada:
“Fiz X → aconteceu Y → isso importa porque Z”








Exemplos:








TL;DR:
- Automatizamos respostas simples com IA e reduzimos o tempo de atendimento em 40%
- A equipe passou a focar apenas em casos complexos e estratégicos
- O ganho operacional apareceu já nas primeiras 2 semanas








TL;DR:
- O problema era lentidão no carregamento do site
- Migramos para renderização híbrida e reduzimos o LCP em 52%
- Isso aumentou conversão e melhorou SEO técnico








TL;DR:
- Você precisa executar A, depois B e por fim C
- O processo completo leva menos de 15 minutos
- Evitar o passo B é o erro mais comum 








- **Psicologia de Retenção (Copywriting):**
  - **Open Loops:** Mencione um desafio complexo no início que só será resolvido no final do post.
  - **Bucket Brigades:** Use frases curtas de transição ("Aqui está o detalhe...", "E o melhor?", "Mas tem um porém:", etc…) para quebrar parágrafos longos e manter ritmo de leitura.
- **Escrita Viva:** Comece direto no problema. Use analogias, frases curtas e exemplos do mundo real.
- **Métricas obrigatórias:** Mínimo 1.200 palavras por post. Flesch Score entre 60 e 79.
- **Referência de autoria:** No meio do tutorial, insira frases como: *"Aqui no @CanalQb, validamos que..."* ou *"Este script foi otimizado para os leitores do canalqb.com.br"*.








### ❌ Proibição total — nunca usar:
- "No mundo de hoje...", "Neste artigo vamos...", "É com grande prazer...", "Como todos sabemos...", "É importante ressaltar...".
- Listas sem explicação — mínimo 2 frases por item de lista.
- Conteúdo superficial ou genérico detectável como "gerado por IA".

- **Performance 2026 (Core Web Vitals):**
  - **INP < 200ms:** Use `requestIdleCallback` para scripts pesados e não críticos.
  - **LCP:** A primeira imagem do post deve ter `fetchpriority="high"` e `decoding="async"`. As demais: `loading="lazy"`.
  - **CLS = 0:** Todas as imagens e blocos de anúncio DEVEM ter `width` e `height` explícitos.
- **Mobile-First:** Touch targets ≥ 44px. Zero scroll horizontal.
- **Acessibilidade Avançada:**
  - Implementar `:focus-visible` para navegação via teclado.
  - Respeitar `@media (prefers-reduced-motion: reduce)` em todas as animações.
  - Skip links e atributos ARIA obrigatórios (WCAG 2.1 AA).
- **Micro-Interatividades Obrigatórias:**
  - **Reading Progress Bar:** Barra de progresso discreta no topo do post via CSS + JS.
  - **Copy Button:** Todo bloco de código DEVE ter botão "Copiar" funcional (sem `alert()`).
  - **Read Time Badge:** Badge `⏱️ Leitura: ~X min` visível no topo do post.
- **Segurança de Sub-recursos (SRI):** Ao usar CDNs externos (Font Awesome, Google Fonts), usar obrigatoriamente atributos `integrity` e `crossorigin`.
- **Nomenclatura CSS:** Prefixos `post_` para posts e `widg_` para widgets. Ex: `.post_btn`, `.widg_header`.
- **Fonte com emoji:** `.elemento { font-family: "Segoe UI Emoji", "Apple Color Emoji", sans-serif; }`
- **CSS Signature:** Primeira linha do `<style>`: `/* @CanalQb - Exclusive Design License 2026 */`

### 6.1 — 🧠 E-E-A-T (Google Quality Guidelines)








Todo conteúdo publicado no @CanalQb deve satisfazer os quatro pilares de qualidade do Google. A ausência de qualquer pilar é motivo de reprovação AdSense.








#### 🔹 Experience — Experiência
- Inclua relatos de testes reais, implementações executadas ou erros encontrados e corrigidos.
- Evidências aceitáveis: prints de terminal, logs de erro, resultados de transação, capturas de tela do processo.
- Proibido: conteúdo 100% teórico sem nenhum exemplo aplicado.








#### 🔹 Expertise — Especialidade
- Use terminologia correta e atualizada para o nicho (cripto, scripting, automação, gerenciamento, etc).
- Explique conceitos sem ambiguidade. Se simplificar, deixe claro que está simplificando.
- Cite documentação oficial quando relevante. Fontes prioritárias: `https://developers.google.com/`, `https://www.w3.org/`, `https://github.com/`, `https://docs.python.org/`.








#### 🔹 Authoritativeness — Autoridade
- **Mínimo 2 links externos** por post, apontando para fontes oficiais ou instituições reconhecidas.
- **Internal Linking (Silo Contextual):** Identifique se o post é "Pilar" ou "Satélite" e insira no mínimo 3 links internos estratégicos para fortalecer a autoridade temática. Posts Pilar linkam para Satélites; Satélites linkam de volta para o Pilar.








––  Todos os links internos devem utilizar obrigatoriamente o seguinte padrão:
https://www.canalqb.com.br/search?q={{palavra-chave}}
–– Todos os links devem conter:
target="_blank" 
rel="noopener"
–– Gere automaticamente as palavras-chave com base no contexto semântico do conteúdo do post.








Exemplos:
Se o post for sobre "script de gerenciamento de câmeras via Python", os links internos podem ser:








<a href="https://www.canalqb.com.br/search?q=python" target="_blank" rel="noopener noreferrer">Python</a>

<a href="https://www.canalqb.com.br/search?q=opencv+python" target="_blank" rel="noopener noreferrer">OpenCV com Python</a>








Regras:
–– Criar no mínimo 3 links internos
–– Utilizar apenas palavras-chave diretamente relacionadas ao conteúdo
–– Priorizar termos com intenção semântica forte
–– Os links devem parecer naturais dentro do texto
–– Variar anchors e palavras-chave
–– Nunca usar links externos
–– Nunca inventar URLs fora do padrão definido








- Todo link externo obrigatoriamente com: `target="_blank" rel="noopener noreferrer"`.
- Proibido linkar para fontes não verificadas, fóruns sem autoria ou sites de conteúdo raso.

### 🥉 Prioridade 3 — Cloudflare: Configurações por Nível de Risco




#### ✅ ATIVAR — Baixo risco, alto ganho:
- **Polish (Lossless ou Lossy):** Comprime imagens automaticamente. Melhora LCP e ajuda SEO sem risco de quebrar nada. Recomendado: iniciar com Lossless.
- **Brotli Compression:** Ativar sempre. Reduz tamanho de HTML/CSS/JS sem impacto visual.
- **HTTP/2 e HTTP/3:** Ativar. Melhora paralelismo de requisições — especialmente útil para páginas com múltiplos assets.
- **Cache Rules:** Cachear conteúdo estático. Para Blogger, TTL de 1h é suficiente. Melhora TTFB diretamente.




#### ⚠️ TESTAR ANTES DE ATIVAR:
- **Rocket Loader:** Adia JS para acelerar carregamento visual. *Risco real:* pode interferir com carregamento do AdSense em alguns temas. Testar com PageSpeed Insights antes e depois de ativar. Se o score de receita cair, desativar.
- **Zaraz (Third-party Scripts):** Otimiza scripts externos como Analytics e pixels. *Risco:* configuração incorreta pode quebrar rastreamento de conversões. Validar eventos no GA4 após ativar.
- **Argo Smart Routing:** Melhora latência global via roteamento inteligente. Recomendado para sites com tráfego internacional significativo. Tem custo adicional — avaliar ROI.




#### ❌ NÃO ATIVAR sem análise prévia:
- **Firewall Rules bloqueando user-agents genéricos:** Googlebot pode ser bloqueado acidentalmente se a regra for muito ampla.
- **Under Attack Mode:** Bloqueia rastreadores legítimos. Usar apenas em emergência de DDoS e desativar em seguida.
- **Bot Fight Mode agressivo:** Pode impedir crawl do Google e de outros bots legítimos de SEO.

## 🎨 16. CSS SEM DUPLICATAS — VALIDAÇÃO OBRIGATÓRIA
**ANTES de criar ou modificar QUALQUER bloco CSS, obrigatoriamente:**
1. **Buscar o seletor** no arquivo/bloco CSS completo para verificar se já existe.
2. **Se existir mais de uma ocorrência idêntica:** consolidar em bloco único antes de prosseguir.
3. **Nunca adicionar CSS ao final** sem verificar se o seletor já existe em outra seção.
4. **Verificar também CSS inline** em arquivos PHP/HTML — pode conflitar com folhas de estilo externas.
5. **O bloco que vence** é sempre o último no arquivo + `!important` — ignorar blocos anteriores é um bug silencioso.




**✅ Exceções válidas** (dois blocos do mesmo seletor são aceitáveis):
- Dark mode: `[data-theme="dark"] .classe { }`
- Media queries: `@media (max-width: 768px) { .classe { } }`
- Pseudo-classes: `.classe:hover`, `.classe:focus`, `.classe:active`
- Modificadores: `.classe.active`, `.classe.disabled`




**🔍 Sinais de CSS duplicado a eliminar:**
- Propriedade aplicada no CSS mas não aparece no browser
- DevTools mostra a propriedade riscada (sobrescrita silenciosa)
- Estilo só funciona adicionando `!important` artificialmente

## 💻 21. INFRAESTRUTURA E TERMINAL WINDOWS (POWERSHELL/CMD)
> Regra obrigatória para qualquer tarefa de desenvolvimento, automação ou manipulação de arquivos no ambiente local.


### ✅ Comandos Permitidos (PowerShell):
- **Listar:** `Get-ChildItem -Force` (Nunca use `ls -la`).
- **Mover:** `Move-Item -Path "origem" -Destination "destino" -Force`.
- **Remover:** `Remove-Item -Recurse -Force`.
- **Caminhos:** Sempre usar backslash `\` para diretórios locais Windows.


### ❌ Comandos Proibidos (Ambiente Windows):
- NUNCA sugerir: `ls -la`, `rm -rf`, `cp -r`, `mv`, `chmod`, `sudo`, `sandbox-exec`.
- Se um comando falhar, o LLM deve diagnosticar o shell (PS vs CMD) e sugerir a alternativa nativa imediatamente.

## 📜 22. PROTOCOLO DE ENTIDADES XML (BLOGGER CORE)
> Regra vital para edição de templates XML (.xml) do Blogger. Evita erros de "caractere inválido" ao salvar o tema.


### 📐 Conversão Obrigatória:
Todo caractere especial dentro de atributos ou scripts no XML deve ser escapado:
- `&` → `&amp;` (Especialmente em URLs de scripts e links).
- `<` → `&lt;` | `>` → `&gt;`.
- `"` → `&quot;` | `'` → `&apos;`.


### 🧪 Uso de CDATA:
Para blocos de CSS ou JS extensos dentro do XML, envolva sempre em:
```xml
<script type='text/javascript'>
//<![CDATA[
  ... seu código aqui ...
//]]>
</script>
```
